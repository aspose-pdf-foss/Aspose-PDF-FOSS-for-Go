## API design

When building new functionality, always shape the public API to mirror **Aspose.PDF for .NET**. Its class/method names, types, and call patterns are the reference model: prefer the same concepts (e.g. `Document`, `Page`, `TextFragment`, `Table`/`Row`/`Cell`, `OutlineItemCollection`, `BorderInfo`/`MarginInfo`), the same method names, and the same overall workflow, adapting only where Go idioms require it (error returns instead of exceptions, `io.Reader`/`io.Writer` stream variants, exported funcs over constructors). A Go developer who knows Aspose.PDF for .NET should recognize this API immediately. When a new feature has a counterpart in Aspose.PDF for .NET, check that API first and align with it before inventing a new shape; the existing surface (noted as "Mirrors Aspose.PDF for .NET's ŌĆ”" throughout the Public API section below) shows the pattern to follow.

## Architecture

Pure Go library. No external dependencies. All PDF functionality is in the root package `asposepdf`; AI-powered operations live in the `ai` subpackage (stdlib-only too ŌĆö `net/http` against OpenAI-compatible endpoints), so the root package stays free of AI and network code.

### Public API

- `(*Document).PageCount()` ŌĆö current page count

- `(*Document).SetPermissions(p Permissions)` ŌĆö configures viewer-enforced permissions (print, copy, modify, etc.) for encrypted documents; applied on Save/WriteTo

- `(*Document).SearchText(query, opts...) ([]TextMatch, error)` ŌĆö finds occurrences of query across all pages (each match carries its page); mirrors Aspose.PDF for .NET's `TextFragmentAbsorber`
- `(*Document).ReplaceText(old, replacement, opts...) (int, error)` / `(*Page).ReplaceText(...)` ŌĆö replaces every occurrence and returns the count; redraws the replacement at the same baseline/size/colour in a metric-compatible Standard-14 face (so any text renders even over an embedded subset font); no line re-flow; mirrors the find-and-replace idiom of Aspose.PDF for .NET's `TextFragmentAbsorber` + `TextFragment.Text` (`replace.go`)

**`form_xobject.go`** ŌĆö reusable Form XObjects (ISO 32000-1 ┬¦8.10): draw content once, place it on any number of pages/positions with a single `Do`. Mirrors the intent of Aspose.PDF for .NET's `XForm`
- `(*Document).CreateForm(width, height) *XForm` ŌĆö an empty Form XObject of the given size. `(*XForm).Canvas() *Page` returns the **drawing surface** ŌĆö a detached page (not in `doc.pages`) on which the full `*Page` drawing API works (`AddText`, `Draw*`, `AddImage`, `AddTable`, ŌĆ”); `(*XForm).Size()` returns its dimensions
- `(*Page).AddForm(form *XForm, rect) error` ŌĆö places the form so its `/BBox` maps onto `rect` (per-axis scale + translate via a `cm`). Placement reuses `registerFormXObject` (`/Fm0`, `/Fm1`, ŌĆ” in the page `/Resources/XObject`); `bboxFromForm` reads a parsed form's `/BBox`. **`imposition.go`** ŌĆö N-up / booklet page imposition (returns a new `*Document`; receiver untouched). - `(*Document).Booklet(opts BookletOptions) (*Document, error)` ŌĆö imposes pages two-up reordered for saddle-stitch (print double-sided, fold, read in order); page count padded up to a multiple of 4 with blank halves

- Internals: `importPageAsXObject` wraps a page (content + `/Resources`) as a `/Subtype /Form` XObject with `/BBox` = MediaBox; `importGraph` copies each page's resource graph into the result once (shared objects deduped via a single `idMap`, reusing `collectValueDepsDoc` + `rewriteRefs`); `makePlacement` computes the fit-inside `cm`; `addImposedSheet` emits `q cm /Xn Do Q` per cell. - `(*Document).SubsetFonts() (int, error)` ŌĆö rebuilds every embedded TTF (`LoadFont`) to keep only the glyphs actually drawn, returning the number of fonts subsetted. Call once, after all text is added and before Save/WriteTo (later text could reference dropped glyphs). Typical reduction ~300-700 KB/font ŌåÆ a few KB. Implementation (`font_subset.go`): tracks used glyph IDs as the text encoders emit them, takes the transitive closure over composite-glyph components, renumbers the kept glyphs into a compact range, rebuilds `glyf`/`loca`/`hmtx`/`maxp`/`head`/`hhea` + a synthetic `post` 3.0 (drops `cmap`/`OS/2`/`name`/hinting), and reassembles the sfnt with correct checksums. Original glyph IDs are preserved as CIDs via a generated `/CIDToGIDMap` stream (was `/Identity`), so no content stream is rewritten; `/W` and `/ToUnicode` are trimmed to the used glyphs. Mirrors the post-processing shape of `OptimizeImages`

**`encrypt.go` / `decrypt.go` / `encrypt_aes.go` / `decrypt_aes.go` / `encrypt_aes256.go` / `decrypt_aes256.go`**
- `Encrypt(inputPath, outputPath, userPassword, ownerPassword)` ŌĆö top-level helper writes RC4-128-protected PDF (PDF 1.4 Standard Security Handler V=2 R=3). For AES, use `(*Document).SetEncryption(EncryptionOptions{...})`

- Decryption pipeline: `OpenWithPassword`/`OpenStreamWithPassword` parse `/Encrypt`, dispatch by `/V` (V=2 R=3 ŌåÆ RC4 path; V=4 R=4 ŌåÆ AES-128 path via `/CFM /AESV2`; V=5 R=6 ŌåÆ AES-256 path via `/CFM /AESV3` per ISO 32000-2). **`sign.go` / `pkcs7_sign.go` / `pkcs7_timestamp.go` / `sign_incremental.go` / `sign_verify.go` / `pkcs7_verify.go` / `appearance_sign.go`** ŌĆö digital signatures (ISO 32000-1 ┬¦12.8; epic `pdf-go-bm9`). - `SignOptions.TimestampURL string` ŌĆö fetch an RFC 3161 timestamp from that TSA over the signature value and embed it as the signature-time-stamp **unsigned** attribute ([1] IMPLICIT SET), anchoring the time to a trusted clock (`pkcs7_timestamp.go`: `buildTimestampRequest` ŌåÆ POST `application/timestamp-query` ŌåÆ parse `TimeStampResp` ŌåÆ `marshalTimestampUnsignedAttr`). - `SignOptions.Incremental bool` + **multiple signatures** (`sign_incremental.go`) ŌĆö signing writes an incremental update (the original bytes verbatim + an appended revision: new/modified objects, a classic xref section with `/Prev` ŌåÆ the previous one, a fresh trailer carrying the original `/ID`) instead of rewriting, so any earlier signature stays valid. - **Signing encrypted documents** ŌĆö an encrypted document is always signed incrementally: the appended signature objects are encrypted with the document's own scheme (RC4-128 / AES-128 / AES-256) while the signature `/Contents` and `/ByteRange` stay plaintext (written as `pdfRaw`, per ISO 32000-1 ┬¦7.6.2). The reader side exempts a signature dict's `/Contents` from decryption (`isSignatureDict` in `decrypt.go`, applied across the RC4/AES-128/AES-256 decrypt walkers). A freshly-built encrypted document (no source) is signed by `buildFreshEncryptedSignedPDF`: serialize the encrypted PDF, reopen it (so it has source + preserved encryption), then sign incrementally ŌĆö the only form pyHanko/Acrobat accept (full-rewrite-then-sign confuses revision-aware validators). The incremental trailer repeats `/Encrypt`. Validated with pyHanko across all three algorithms: `INTACT:TRUSTED,UNTOUCHED`, `ENTIRE_FILE`
- Visible signature: `SignOptions.Visible` + `Rect Rectangle` + `Page int` (mirrors Aspose `Sign(..., visible, rect, page)`) draw a generated appearance block ("Digitally signed by ŌĆ”", date, optional reason/location) via `generateSignatureAppearance` (`appearance_sign.go`) ŌåÆ an `/AP/N` Form XObject drawn with a Standard-14 font (so the built-in renderer shows it). `SignOptions.Appearance *SignatureAppearance` customizes content/style (`ShowName`/`ShowDate`/`ShowReason`/`ShowLocation`/`Font`/`FontSize`/`Color`; nil = full default block) ŌĆö mirrors Aspose's `SignatureCustomAppearance`. - `(*Document).VerifySignatures() ([]SignatureVerification, error)` ŌĆö verifies every `/Sig` field against the document's source bytes (stored on Open in `d.source`). `SignatureVerification` ŌĆö `FieldName`, `SignerName`, `Reason`, `Location`, `SigningTime`, `Valid`, `IntegrityOK`, `CoversWholeDocument`, `Certificate *x509.Certificate`, `Err`. Trust-chain validation is the caller's (the cert is returned)
- The byte-range dance (`sign.go`): the writer emits fixed-width `pdfRaw` placeholders for `/Contents` (`<` + 16384 zero hex + `>`) and `/ByteRange` (`[0 0000000000 ŌĆ”]`); `buildSignatureObjects` adds the `/Sig` dict + invisible Sig field/widget (`Rect [0 0 0 0]`) into `/AcroForm/Fields` with `/SigFlags 3`; after serialization `applySignature` finds the placeholders, fills the real `/ByteRange` (the excluded gap is the whole `/Contents` value **including the `<>` brackets** ŌĆö the convention validators expect), hashes the covered bytes, builds the PKCS#7, and splices the hex in-place without shifting offsets
- CMS is hand-rolled on `encoding/asn1` (stdlib has no PKCS#7): `buildPKCS7Detached` (SignedData with contentType/messageDigest/signingTime signed attributes; signature over the SET-OF-re-tagged attrs) and `verifyPKCS7Detached` (parse, recover signer cert by issuer+serial, check messageDigest == SHA-256(content), verify RSA PKCS#1v1.5 / ECDSA). Validated against **OpenSSL** (`cms -verify`) and **pyHanko** (`INTACT:TRUSTED,UNTOUCHED`, coverage `ENTIRE_FILE`) ŌĆö invisible, visible, PAdES, DocMDP, RFC 3161-timestamped, multi-signature, and encrypted-document alike. Out of scope: LTV (DSS/VRI), image/logo in the appearance
- Test certs are generated in-memory (`crypto/x509`); **no private key is ever committed** (avoids secret-scanner findings)

**`flow.go`** ŌĆö flow / document-generator layout (epic `pdf-go-m9kt`, Tier 1): the **flow counterpart to the Rectangle-based drawing API** ŌĆö content is laid out top-to-bottom and paginated automatically, instead of positioned absolutely. An additive layer (it computes Rectangles and calls the existing `AddText`/`AddImage`/`AddTable`/`drawList`), so it coexists with the Rectangle paradigm. Mirrors the intent of Aspose.PDF for .NET's generator / `Page.Paragraphs` flow model
- `(*Document).NewFlow(opts FlowOptions) *Flow` ŌĆö `FlowOptions{Format, Margin{Left,Right,Top,Bottom} (zero ŌåÆ 54pt), ParagraphSpacing (zero ŌåÆ 6pt), Tagged *TaggedContent}`. Builders (chainable): `AddParagraph`, `AddHeading(level, ŌĆ”)`, `AddImage(path, w, h)` / `AddImageAlt(ŌĆ”, alt)` (h Ōēż 0 ŌåÆ preserve aspect via `image.DecodeConfig`), `AddTable(*Table)`, `AddList(items, ordered, style)`, `AddSpacer(h)`. `Render() (pages int, error)` lays everything out and returns the page span
- Engine: a `flowState` cursor (current page + Y) over a content box (page minus margins). `startPage` reuses the document's initial blank page (resized) when it has exactly one with no content, else appends. Paragraphs/headings **split across pages line-by-line** (`wrapText` ŌåÆ draw the lines that fit, new page, continue); images are atomic (scaled down to one content area if oversized); tables draw inline when they fit, on a fresh page otherwise, and fall back to `AddTable`'s own pagination when taller than a page; lists use the shared `drawList`. When `Tagged` is set, each element is auto-tagged (`/P`, `/H1ŌĆ”/H6`, `/Figure`+`/Alt`, `AddTaggedTable`, tagged `drawList`) so the flow output passes `ValidatePDFUA` ŌĆö a one-call accessible report generator
- **Floating boxes** (`flow_box.go`, Tier 2): `FloatingBox` is a positioned content container with an optional border/background/padding holding its own content (`NewFloatingBox().AddParagraph/AddHeading/AddImage/AddList/AddSpacer().SetBorder/SetBackground/SetPadding/SetSpacing`). `SetSpacing(gap)` overrides the inter-element gap (default 6pt; `0` for a single-paragraph box so symmetric padding centres it vertically). The border honours `BorderInfo.Sides` ŌĆö a single side renders as a rule (e.g. `BorderSideLeft` for a block-quote) rather than a full outline (`drawBoxFrame` draws per-side edges, only `BorderSideAll` strokes the whole rect). `(*Page).AddFloatingBox(box, rect)` places it absolutely (paints the frame, flows content within `rect`ŌłÆpadding, clips on overflow); `(*Flow).AddFloatingBox(box)` inserts it in the flow at its **measured** height (`measureFlowElems`; new page if it doesn't fit). The box lays out via a `boxed` `flowState` (`newPage` ŌåÆ `errBoxFull` stops instead of paginating) reusing all element drawing; when the document is tagged the box is a `/Div` (content tagged under it via the new `flowState.parent`) and its frame an `/Artifact`. Tables are not allowed inside a box (they paginate)
- **Multi-column layout** (Tier 3): `FlowOptions.Columns` (>1) + `ColumnGap` (zero ŌåÆ 18pt) divide the content box into N columns; the engine fills a column top-to-bottom then moves to the next via `flowState.advance` (column ŌåÆ next column ŌåÆ next page), with `colLeft()` giving the current column's X and `contentW` the per-column width. `(*Flow).AddColumnBreak()` forces the following content into the next column (or, from the last column, the next page) ŌĆö used to put one section per column. Works with tagging (content stays under the structure tree across columns) and pagination; tables/boxes flow within the current column (a wide table in a narrow column overflows ŌĆö use a single column for those). Boxed (`FloatingBox`) layout stays single-column (`advance` ŌåÆ `errBoxFull`)
- **Text flow-around** (`flow_float.go`, Tier 3): `(*Flow).AddFloatBox(box, FloatLeft|FloatRight, width)` draws a `FloatingBox` pinned to a column edge and registers an `activeFloat` band; the following paragraphs wrap **line by line** around it (`flowText` ŌåÆ `flowTextAround` when `len(floats) > 0`): each line's `[left,right]` comes from `lineBounds(y)` (column minus any active float band + `floatTextGap`), `nextLine` greedily fits words into that width, lines on one page are bracketed into a single tagged `/P` (`flush`), and the cursor returns to full width once it passes the float's `bottom` (`pruneFloats`). A line with no room beside the float drops to `nextFloatBottom`; non-wrapping elements (image/table/list/box) call `dropBelowFloats` to start on a clean line below the float. Floats are cleared on a column/page break (so they don't span pages). Designed for single-column flows; in a multi-column flow the box floats within the current column
- Tier-3 remaining (not yet): keep-with-next / widow-orphan control, image floats (only boxes float), splitting a single paragraph's tagging into one `/P` across pages (currently one `/P` per page-chunk)

### PDF parsing pipeline

1. **`io.go`** ŌĆö file I/O (`readFile`, `writeFile`)
2. **`xref.go`** ŌĆö locates and parses the cross-reference table or stream; handles both traditional xref tables (PDF Ōēż1.4) and cross-reference streams (PDF 1.5+)
3. **`lexer.go`** ŌĆö byte-level tokenizer; produces tokens (int, float, name, string, keyword, etc.)
4. **`parser.go`** ŌĆö builds `pdfValue` objects from tokens; handles dicts, arrays, streams with FlateDecode/LZWDecode (`lzw.go`, MSB-first 9ŌĆō12-bit codes, `/EarlyChange` 0 and 1)/ASCIIHex/ASCII85/RunLength filters and PNG predictor (Predictor 12). CCITTFaxDecode (`ccitt.go` / `ccitt_tables.go`) decodes Group 4 (`/K<0`) and Group 3 1-D (`/K=0`) fax data into packed 1-bpp rows (used by 1-bit scanned images and image masks); reads `/Columns`/`/Rows`/`/BlackIs1`/`/EncodedByteAlign` from `/DecodeParms` (mixed Group 3 2-D, `/K>0`, is not yet supported)
5. **`doc.go`** ŌĆö document-level logic: object lookup with caching, object streams (ObjStm), page tree traversal, dependency collection
6. **`types.go`** ŌĆö type definitions: `pdfValue`, `pdfDict`, `pdfArray`, `pdfStream`, `pdfRef`, `pdfObject`, `xrefEntry`

**Malformed-input tolerance** (`openStreamCore` ŌåÆ `buildFromXRef`, with fallbacks):
- Stream `/Length` may be an indirect reference (`/Length N 0 R`, ISO 32000-1 ┬¦7.3.8.2), missing, or wrong ŌĆö `readStreamData` (`parser.go`) takes a usable direct integer verbatim, otherwise scans for the `endstream` keyword via `streamEndIndex`, which prefers the `endstream` confirmed by a following `endobj` so a spurious `endstream` byte sequence inside binary stream data (common in CCITT/JPEG streams) doesn't truncate the stream mid-data.
- `skipToStreamData` (`lexer.go`) skips spaces/tabs before the CR/LF after the `stream` keyword (non-conformant `stream \r\n`), so the filter sees a correctly aligned data start instead of mis-decoding (which previously caused a multi-GB allocation when the resulting garbage parsed as content ops).
- `flateDecode` (`parser.go`) keeps the inflated bytes when a valid-header zlib stream ends in a bad trailing Adler-32 checksum or is truncated by a wrong `/Length` (common in the wild ŌĆö otherwise the whole page blanks). A bad zlib *header* is still rejected, so an undecrypted/random stream is not inflated into garbage.
- xref entry rows are read up to any line terminator (LF, CR, or CRLF), so classic-Mac CR-only xref tables parse.
- `xref_reconstruct.go` ŌĆö when the xref is missing/corrupt or the catalog/page tree won't resolve (e.g. an off-by-one subsection start), `reconstructXRef` rebuilds the table by scanning the file for `N G obj` headers (latest revision wins) and recovers the trailer from the last `trailer` dict or the first `/Type /Catalog` object, then `openStreamCore` retries. Limitation: objects only inside compressed object streams (ObjStm) aren't recovered by the scan.

### PDF writing (`writer.go`)

`buildDocumentPDF(d *Document)` is the sole output function:
1. Assign sequential output IDs to all objects in `d.objects`
2. Patch `/Parent` in every page dict to point to the new `/Pages` node (via `pdfDirectRef`)
3. Serialize each object; write `/Pages`, `/Catalog`, `/Info`, `/Encrypt` structural objects last
4. Write xref table + trailer

**`pdfDirectRef`** (defined in `types.go`) ŌĆö like `pdfRef` but written by `writeValue` without remapping. Used for `/Parent` patches so that the new `/Pages` object number (output space) is never accidentally remapped.

**Object assembly is shared** (`linearize_assemble.go`): `(*Document).assemble()` builds the output object set + structural metadata (sequential IDs, the `/Pages` node, output `/Catalog`, `/Info`, `/Encrypt` IDs, encryption state, file header) and is consumed by **both** `buildDocumentPDF` (sequential layout) and `buildLinearizedPDF` (linearized layout), so numbering and structural-object construction live in exactly one place.

### Linearization / fast web view (`linearize.go` / `linearize_emit.go` / `linearize_hint.go` / `linearize_assemble.go`)

`(*Document).SaveLinearized(path)` / `WriteToLinearized(w)` write a **linearized** ("fast web view") PDF per ISO 32000-1 Annex F (epic `pdf-go-h7r`): a physical layout optimised for streaming, where the first page's objects and a primary hint table are reachable from the front of the file so a viewer can render page 1 before the whole file downloads. The output is an ordinary, fully spec-compliant PDF that any reader opens normally (linearized files are also valid non-linearized files). Encryption and signing are not supported in combination with linearization (rare for web-served PDFs); such documents return an error. Mirrors the intent of Aspose.PDF for .NET's linearized save.
- Layout (classic cross-reference tables): header ŌåÆ linearization parameter dict (`/Linearized /L /H /O /E /N /T`, first object, within the first 1024 bytes) ŌåÆ first-page xref + trailer (`/Prev` ŌåÆ main xref, early) ŌåÆ catalog ŌåÆ primary hint stream ŌåÆ first page's private objects ŌåÆ shared objects ŌåÆ body (pages 2..N, page-tree, orphans, info) ŌåÆ main xref + trailer ŌåÆ `startxref` (ŌåÆ first-page xref). Objects are renumbered part 2 = 1..K2 (body, low), part 1 = K2+1.. (first-page section, high), matching qpdf's convention.
- Partitioning (`firstUse` model): every object is attributed to the first page that references it; the first page owns all the objects it uses, each later page owns only the objects no earlier page used. Objects referenced by more than one page are *shared*; their `nObjects` count is attributed to the first page while their bytes sit at the end of the first-page section. The primary hint stream carries the **page-offset hint table** (per-page object count, length, content-stream length, shared references ŌĆö bit-packed, 13-field header) and the **shared-object hint table**, both in the column-major field layout qpdf reads. Two-pass-free: hint-table bit widths depend only on object/page/content *lengths* (offset-independent), so objects are serialized once to obtain lengths, the hint structure is computed, then absolute offsets are filled in.
- **Validation status**: qpdf 12.3.2 `check_linearization` passes for documents **without cross-page shared objects** (single-page and multi-page documents whose pages don't share fonts/resources). For documents that *do* share objects across pages, the output is still a valid, readable, `is_linearized`-true fast-web-view PDF (Acrobat shows "Fast Web View: Yes"), but qpdf's strict hint-table recompute is not yet matched ŌĆö the per-page `pageLen` accounting for shared-resource pages follows an undocumented qpdf-internal formula still being reverse-engineered (tracked under `pdf-go-h7r`). The validation harness (pikepdf `Job(['qpdf','--check'])` with field-level diagnostics) lives in `result_files/lin/` (gitignored).

### Dependency collection (`doc.go`)

`collectPageDeps` recursively walks the object graph (dict values, array elements, stream dict, and raw stream bytes via regex `\b(\d+)\s+\d+\s+R\b`) to find all objects needed for a page. Skips `/Pages` and `/Catalog` nodes ŌĆö these are rebuilt by the writer. Used by `Split` and `Extract` to build new single-document object sets.

### Text extraction (`text.go`, `text_layout.go`, `content_parser.go`, `font.go`, `font_metrics.go`, `encoding.go`, `cmap.go`)

1. `parseContentStream(data)` tokenizes content stream bytes into `contentOp` structs (operator + operands), reusing the existing `lexer`
2. `resolveFont(objects, fontDict)` maps font dictionaries to `fontInfo` ŌĆö supports WinAnsi, MacRoman, MacExpert, Standard encodings, `/Differences`, standard 14 fonts, Symbol, ZapfDingbats, ToUnicode CMap, Type0/CIDFont with Identity-H encoding, and **predefined CJK CMaps** (see the CJK section below ŌĆö `resolveType0Encoding` resolves a named `/Encoding` like `/GBK-EUC-H` to a codeŌåÆCID `cidCMap` and the descendant's `/Ordering` to a CIDŌåÆUnicode table); resolves glyph widths from `/Widths`, Standard 14 metrics, CID `/DW`+`/W`, or fallback
3. `parseCMap(data)` (`cmap.go`) parses ToUnicode CMap streams ŌĆö handles `beginbfchar`/`endbfchar` and `beginbfrange`/`endbfrange` (sequential and array forms); returns `map[uint16]rune`
4. `textExtractor` state machine processes operators (BT/ET/Tf/Td/Tm/Tj/TJ/Tz/etc.), tracking text matrix position, font, spacing, and horizontal scaling; advances text matrix by glyph width after each character (PDF spec 9.4.4); splits into single-byte and multi-byte paths for Type0/CIDFont
5. Fragment collection: `emitRune` collects `textFragment` structs with (x, y, endX, fontName, fontSize); new fragment on font change, Y gap > fontSize├Ś0.5, or X gap > spaceWidth├Ś0.3
6. Visual sorting (`text_layout.go`): `groupFragmentsIntoLines` sorts fragments by Y descending then X ascending, groups by Y proximity into `TextLine` structs; `ExtractTextWithLayout` returns the structured result; `ExtractText` delegates to same pipeline
7. Form XObjects (`Do` operator) are recursively processed with inherited CTM and overridden resources
8. Marked content (`BDC`/`BMC`/`EMC`): when `BDC` carries `/ActualText` in its properties, glyph emission is suppressed and the replacement text is emitted at `EMC`; supports inline dicts, `/Properties` resource lookup, UTF-16BE strings, and nesting

### Image extraction (`image.go`, `image_decode.go`, `image_inline.go`)

1. Content stream walker tracks CTM via `cm`/`q`/`Q` and collects images on `Do` (XObject) and `BI` (inline)
2. DCTDecode images are passed through as JPEG; a chained `/Filter` array ending in the image codec (e.g. `[/FlateDecode /DCTDecode]`, common in scanned MRC PDFs) is unwrapped by applying the preceding filters first (`filterChain` + `unwrapped` in `extractXObjectImageData`); **CMYK** DCTDecode (and any with an `/SMask` or a stencil `/Mask`) are decoded to RGB and re-encoded as PNG, because Adobe/Photoshop CMYK & YCCK JPEGs store their channels inverted (APP14 "Adobe" marker) and Go's `image/jpeg` returns them raw ŌĆö `jpegHasAdobeMarker` detects the marker and `decodeJPEGToPixels` re-inverts so white doesn't render as black; all non-DCT images are decoded to pixels and encoded as PNG
3. Colour-key masking (`/Mask` as a sample-range array, ISO 32000-1 ┬¦8.9.6.4) is applied as per-pixel alpha computed on the pre-conversion samples (`colourKeyAlpha`) ŌĆö palette indices for Indexed images. Color spaces: DeviceRGB, DeviceGray, DeviceCMYK (ŌåÆRGB via `adobeCMYKToRGB`, a 5├Ś5├Ś5├Ś5 LUT baked from the MuPDF/Adobe default CMYK profile with quadrilinear interpolation ŌĆö `cmyk_lut.go` ŌĆö so process colours match Acrobat instead of the bluish naive `(1-C)(1-K)`; used by image decoding and the renderer's `k`/`K`/`scn` fill paths), Indexed (palette expansion), ICCBased (treated as underlying RGB/Gray/CMYK)
4. Soft masks (`/SMask`) are applied as PNG alpha channels (`decodeSoftMaskData`: JBIG2-coded masks, 1/2/4-bpc expansion, `/Decode [1 0]` inversion; `fitSoftMask` reconciles a mask whose resolution differs from the image ŌĆö a higher-res bilevel text mask upscales the colour to the mask grid, MRC-style); JPEG+SMask is re-encoded as PNG
5. Inline images (BI/ID/EI) are parsed with abbreviation expansion (PDF spec Tables 4.43/4.44)
6. Form XObjects are recursed into with inherited CTM and resources
7. **JBIG2** (`/JBIG2Decode`, 1-bpp scanned bilevel): decoded in-house (`jbig2*.go`) since `decodeStream` can't reach the `/JBIG2Globals` stream ŌĆö `extractXObjectImageData` detects the filter, resolves globals from `/DecodeParms` (`jbig2GlobalsData`), and calls `jbig2Decode` to a packed 1-bpp DeviceGray bitmap (0 = black; JBIG2 foreground is inverted), so extraction and the renderer (which shares `Extract`) both work. See the JBIG2 section below for scope.
8. **JPEG2000** (`/JPXDecode`, 8-bpp colour scanned layers): decoded in-house (`jpx*.go`) to interleaved 8-bit RGB/Gray pixels and encoded as PNG. `extractXObjectImageData` detects the filter and calls `jpxDecode`. **MRC stencil masks**: a colour `/JPXDecode` image with an explicit `/Mask` referencing a (usually high-res, JBIG2-coded) `/ImageMask` is composited at the mask's resolution (`decodeStencilMask` + `applyStencilMask`, ISO 32000-1 ┬¦8.9.6.3) so the bilevel text mask stays sharp over the low-res colour foreground ŌĆö the standard scanned-document layering of `/bg` (background) + masked `/fg` (foreground). See the JPEG2000 section below for scope.

### JBIG2 decoding (`jbig2.go`, `jbig2_mq.go`, `jbig2_generic.go`, `jbig2_refine.go`, `jbig2_symbol.go`, `jbig2_text.go`, `jbig2_tables.go`)

Pure-Go JBIG2 decoder (ITU-T T.88) for the PDF `/JBIG2Decode` filter. Epic `pdf-go-hqr`. **Phase 1 (complete)** covers the embedded-organization segment stream on the arithmetic (MQ) coding path ŌĆö the combination scanned-document PDFs almost always use:
- `jbig2_tables.go` ŌĆö the MQ probability-estimation state table (Table E.1).
- `jbig2_mq.go` ŌĆö the MQ arithmetic decoder (software conventions: C split into chigh/clow), the arithmetic integer decoding procedures (`decodeInt` for IADH/IADW/IAEX/IAAI/IADT/IAFS/IADS/IAIT/IARI/IARDW/IARDH/IARDX/IARDY) and `decodeIAID` for symbol IDs. Faithful port of the reference decoder; validated byte-identical to jbig2dec / PyMuPDF.
- `jbig2_generic.go` ŌĆö `jbig2Bitmap` (height├Świdth, 1 byte/pixel) and generic-region decoding (templates 0ŌĆō3 with adaptive pixels and TPGDON typical prediction).
- `jbig2_refine.go` ŌĆö generic refinement-region decoding (templates 0/1), used by refinement/aggregate symbol dictionaries and refining text regions.
- `jbig2_symbol.go` ŌĆö `jbig2Ctx` (all IAx contexts + GB/GR context arrays + IAID), symbol-dictionary decoding by height class; the SDREFAGG path refines a single referenced symbol (REFAGGNINST==1) or aggregates several via a refining text region (>1); IAEX export run-lengths.
- `jbig2_text.go` ŌĆö text-region decoding: strips of symbol instances placed by IADT/IAIT (T), IAFS/IADS (S) and IAID, honoring REFCORNER/TRANSPOSED/SBDSOFFSET and per-instance refinement; the strip loop reads the terminating OOB IADS so the bit stream stays in sync.
- `jbig2.go` ŌĆö segment-header parser (embedded organization), page composition across segments (page info / generic region / symbol dict / text region), and the public `jbig2Decode(stream, globals, w, h)` returning packed 1-bpp. Unknown-length segments (T.88 ┬¦7.2.7, dataLength 0xFFFFFFFF on an immediate generic region) are delimited by scanning for the terminator (0xFF 0xAC arithmetic / 0x00 0x00 MMR) followed by the 4-byte declared height ŌĆö same approach as pdf.js.

**Phase 2 (added: `jbig2_huffman.go`, `jbig2_huffsym.go`, `jbig2_hufftext.go`, `jbig2_mmr.go`, `jbig2_halftone.go`):**
- `jbig2_huffman.go` ŌĆö Huffman bit reader, canonical-prefix table builder/decoder, and the 15 standard tables (Annex B); custom table segments (type 53) parse in `jbig2.go`.
- `jbig2_huffsym.go` ŌĆö Huffman symbol dictionaries: height classes with a collective bitmap (MMR-coded, via `jbig2_mmr.go`, or uncompressed) split into per-symbol bitmaps. `jbig2_hufftext.go` ŌĆö Huffman text regions with the run-code symbol-ID table (┬¦7.4.3.1.7), including transposed regions with SBSTRIPS>1. Validated byte-identical (0 differing pixels) on a real two-page globals + symbol-dict + text-region scan (one page transposed, one not).
- `jbig2_mmr.go` ŌĆö MMR (Group 4) regions/bitplanes reuse the `ccitt.go` G4 decoder (`pdf-go-d6n`).
- `jbig2_halftone.go` ŌĆö pattern dictionaries (type 16) and halftone regions (types 20/22/23): Gray-coded bitplanes select indexed patterns stamped on a grid (`pdf-go-nkf`). Standalone refinement regions (types 40/42/43, `pdf-go-2mk`) decode in `jbig2.go` via the Phase 1 refinement decoder.

Halftone/standalone-refinement are implemented per spec but not yet corpus-validated. Unsupported constructs are skipped, so a page still renders rather than erroring.

### JPEG2000 decoding (`jpx.go`, `jpx_tier2.go`, `jpx_tagtree.go`, `jpx_tier1.go`, `jpx_dwt.go`)

Pure-Go JPEG2000 decoder (ISO/IEC 15444-1) for the PDF `/JPXDecode` filter. Epic `pdf-go-8ju`. Decodes a JP2/raw codestream to interleaved 8-bit samples; validated **pixel-exact** against pdf.js (per-component pre-transform meanAbsDiff 0.0000; final RGB within ┬▒1 on a handful of pixels, ICT floatŌåÆint rounding only) on the `/bg` and `/fg` layers of a real MRC scan.
- `jpx.go` ŌĆö JP2 box walk (`jpxParseBoxes`) and codestream marker parser (`jpxParseCodestream`: SIZ/COD/COC/QCD/QCC/SOT/SOD), the public `jpxDecode(data) (pixels, comps, w, h, err)`. Unsupported COD options (selective arithmetic bypass, per-pass termination, vertically-causal context, predictable termination) surface as `jpxErr` rather than mis-decoding.
- `jpx_tier2.go` ŌĆö tile/precinct/code-block geometry, packet iteration (LRCP/RLCP progression with an LRCP fallback), and packet-header parsing (inclusion + zero-bit-plane tag trees, Lblock length signalling). Subband gain `{LL:0, LH:1, HL:1, HH:2}`.
- `jpx_tagtree.go` ŌĆö the inclusion and zero-bit-plane tag trees (┬¦B.10.2). **The level count is `ceil(log2(max(w,h)))+1`** ŌĆö matching pdf.js exactly. Floor-vs-ceil only diverges for a non-power-of-two code-block grid (e.g. 3 code-blocks across), where floor gives too few tree levels and silently corrupts every higher-resolution subband. This was the decisive bug: low resolutions (1ŌĆō2 code-blocks) matched by coincidence while res Ōēź a 3-wide grid exploded.
- `jpx_tier1.go` ŌĆö EBCOT tier-1 bit-plane decoding (Annex D): the three coding passes (significance propagation / magnitude refinement / cleanup) over each code-block, driven by the shared MQ arithmetic decoder (`jbig2_mq.go` ŌĆö JPEG2000's MQ coder is identical to JBIG2's). 19-entry context state packed as `(stateIndex<<1 | mps)`; context-label tables ported from pdf.js.
- `jpx_dwt.go` ŌĆö inverse quantization, coefficient placement into the resolution grid, inverse DWT (5/3 reversible + 9/7 irreversible lifting), multi-component transform (RCT/ICT) and DC level shift, clamped to 8-bit.

Out of scope (not yet needed by the corpus): tile-parts beyond the first SOT per tile, packed-packet-headers (PPM/PPT), region-of-interest (RGN), and the COD options listed above. Unsupported inputs error cleanly (the image is skipped, the rest of the page still renders).

### CJK / CID-keyed font support (`cmap_cid.go`, `cmap_cid_data.go`, `cjk_cmaps/`)

Non-embedded composite (Type0) CJK fonts ŌĆö e.g. SimSun with `/Encoding /GBK-EUC-H`, Adobe-GB1 ordering, no `/FontFile2` and no `/ToUnicode` ŌĆö previously extracted as U+FFFD and rendered blank. Epic `pdf-go-eaw`. Decisions: **system fonts** (no bundled CJK font ŌĆö those are ~10ŌĆō16 MB) and **all CJK orderings** (Adobe GB1 / CNS1 / Japan1 / Korea1 / KR).
- `cmap_cid.go` ŌĆö `cidCMap`: parses the CMap text format (`begincodespacerange` / `begincidrange` / `begincidchar` / `usecmap`) into a variable-width codespace (1ŌĆō4 bytes) and a codeŌåÆCID lookup (sorted ranges + singles, binary-searched). `next(s)` consumes one code per the codespace (byte-wise containment, ┬¦9.7.6.2), so a single string mixes 1-byte Latin and 2-byte CJK. The same parser serves both the predefined Adobe CMaps and a PDF's own embedded `/Encoding` CMap stream.
- `cjk_cmaps/` + `cmap_cid_data.go` ŌĆö `//go:embed`ed, gzip'd data generated from Adobe's BSD-licensed cmap-resources: the predefined encoding CMaps (codeŌåÆCID, e.g. `GBK-EUC-H`, `90ms-RKSJ-H`, `B5pc-H`, `KSC-EUC-H`, the `Uni*-UCS2/UTF16` families) and a compact per-ordering CIDŌåÆUnicode table (from each `cid2code.txt`'s UTF-16 column, preferring CJK-unified codepoints over radicals/compat). `predefinedCMap(name)` loads + caches a CMap (resolving `usecmap` against the bundle); `cidToUnicodeForOrdering(ordering)` loads the CIDŌåÆUnicode map. ~1.3 MB total.
- `resolveType0Encoding` (`font.go`) wires this into `fontInfo`: `/Encoding` name ŌåÆ `cidCMap`, `/CIDSystemInfo /Ordering` ŌåÆ `cidToUni`. Extraction (`showStringCMap` in `text.go`) decodes codes ŌåÆ CIDs ŌåÆ Unicode (and `/W` widths keyed by CID); Latin codes the CMap maps to proportional-Latin CIDs that Adobe leaves Unicode-less fall back to the ASCII code itself.
- Rendering (`render_text.go`): a non-embedded CJK font resolves to a real outline font via `fontRepo.findCJK` ŌĆö the caller-registered fonts first, then a **separate, lazily-built index of the OS font directories** (kept apart from the opt-in `find()` so Latin substitution stays opt-in), matched by the document's family (SimSun ŌåÆ `simsun.ttc`) or a well-known default family for the ordering (`cjkOrderingFamilies`). Glyphs map CID ŌåÆ Unicode (`cjkUni`) ŌåÆ GID through the system font's cmap; the per-font `cjkASCII` override covers Latin-in-CJK-font runs. With no covering font installed, CJK glyphs are skipped (blank) rather than erroring. A non-embedded *simple* font that is really a CJK face (e.g. SimSun with WinAnsiEncoding for its Latin glyphs) is likewise resolved to the installed CJK font (`isCJKFamily` ŌåÆ `findCJK`) instead of a bundled Latin substitute, so its Latin renders in the document's actual face and stays consistent with its composite (Type0) sibling. Font matching uses an exact PostScript-or-family name and a **compact** (separator-stripped) name (`findSystemExact` / `sysByCompact`), so a BaseFont `YuGothicMedium` resolves to the installed `YuGothic-Medium` / `Yu Gothic Medium` ŌĆö the exact weight, with the correct units-per-em ŌĆö rather than a coarse family default; `findCJK` tries this exact match first, then the ordering-default families (exact, then a word-boundary-prefix fallback `sysFamilyPrefix` so a candidate like `microsoft jhenghei` resolves to the installed `Microsoft JhengHei UI`, while `-ExtB`/`-ExtA` extension faces ŌĆö which carry only rare supplementary ideographs ŌĆö are skipped so common characters don't render blank; e.g. `MSung-Light`/CNS1 ŌåÆ `msjh.ttc`). The CIDŌåÆUnicode table also drives **Identity-H** composite fonts whose descendant is a real Adobe-ordered CIDFont (`/Ordering (Japan1)`, no `/FontFile2` ŌĆö e.g. a non-embedded Yu Gothic): the 2-byte code is the Adobe CID, so CID ŌåÆ Unicode ŌåÆ GID renders the installed face. A non-embedded Identity-H font with no Adobe ordering still falls back to the GID-as-CID path against the real installed font (`findSystemExact`) when present. When that font has an **explicit (non-Identity) `/CIDToGIDMap`** ŌĆö whose GIDs only match the absent embedded program, so GID-as-CID is wrong ŌĆö and a **`/ToUnicode`** CMap with a Latin BaseFont (e.g. Arial in a WinZip/Acrobat visible signature appearance, `pdf-go-5ws`), the renderer substitutes a metric-compatible face (`find` ŌåÆ `findSystemStd14` ŌåÆ bundled `fallbackFontFor`) and reaches glyphs through Unicode: code(=CID) ŌåÆ `/ToUnicode` rune ŌåÆ substitute glyph (reusing the `cjkUni` map, which `gid()` consults before the unusable `/CIDToGIDMap`). The exact installed font still wins for an Identity `/CIDToGIDMap`. Validated: 59015.pdf (GBK-EUC-H / SimSun), plus Japan1 (`90ms-RKSJ-H`, `UniJIS-UCS2-H`) and mixed GB1/CNS1 documents render via the installed CJK fonts; 50466_signed.pdf's visible signature text renders via the ToUnicode-substitute path.

## Output conventions

- All files produced by examples and manual runs are saved to `result_files/` in the project root.
- Generated Markdown reports (API comparisons, release notes, and similar `.md` write-ups) go in `reports/` at the project root instead ŌĆö kept out of `result_files/` so they're easy to find among the binary output.
- Both `result_files/` and `reports/` are not committed to the repository (gitignored).
- Exception: `_examples/feature_showcase/main.go` writes `docs/feature_showcase.pdf` (committed and linked from README). Regenerate only on meaningful changes to keep git history lean.
- Exception: `_examples/showcase_preview/main.go` writes `docs/feature_showcase-preview.png` (committed; the clickable preview in the README) ŌĆö a 2├Ś2 montage of showcase pages rasterized by this library's own `Document.RenderImage`. Regenerate after the showcase changes: `go run ./_examples/feature_showcase` then `go run ./_examples/showcase_preview`.

## Examples

- Standalone runnable example programs live in `_examples/<name>/main.go`. The leading `_` makes the Go toolchain skip these when matching `./...`, so `go build ./...` and `go test ./...` do not touch them. Run individually with e.g. `go run ./_examples/feature_showcase`.
- Short, focused API examples live as `ExampleXxx` functions in `examples_test.go` (package `asposepdf_test`). These appear under "Examples" on pkg.go.dev next to each documented function and are validated by `go test` via their `// Output:` comments.

## Testing conventions

- Test PDF files are stored flat in `testdata/` (`4pages.pdf`, `Binder1.pdf`, `PdfWithLinks.pdf`, `PdfWithTable.pdf`, `alfa.pdf`, `marketing.pdf`, `Hello world.pdf`, `PdfWithAcroForm.pdf`).
- Which files each test uses is declared in `testdata/testfiles.json` ŌĆö keyed by test function name; value is `[][]string` (array of groups, each group is an array of file names). One group = one test run; multiple groups = the test is run once per group.
- When writing tests that use real PDF files, use the `testFile(t)`, `testFiles(t)`, or `testGroups(t)` helpers from `helpers_test.go`, and add the corresponding entry to `testdata/testfiles.json`. Ask the user which file to use before adding a new entry.
- Each feature gets its own `*_test.go` file (e.g. `splitter_test.go`, `metadata_test.go`).
- `TestSplitFiles` in `splitter_test.go` iterates files listed in `testdata/testfiles.json` under `"TestSplitFiles"`, splits each into `result_files/TestSplitFiles/<stem>/`, and validates every output page with `Validate`.
